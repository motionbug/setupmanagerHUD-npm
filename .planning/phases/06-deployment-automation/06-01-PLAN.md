<!-- Existing core repo workflow pattern (reference for structure) -->
From .github/workflows/deploy.yml:
```yaml
name: Deploy to Cloudflare Workers
on:
  workflow_dispatch:
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - run: npm run build
      - run: npx wrangler d1 migrations apply DB --remote
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          CLOUDFLARE_ACCOUNT_ID: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub -> Cloudflare | API token crosses trust boundary from GitHub secrets to Cloudflare API |
| User -> GitHub | User-provided secrets (WEBHOOK_TOKEN) stored in GitHub then deployed to Cloudflare |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | API tokens in logs | mitigate | GitHub automatically masks secrets; avoid echo ${{ secrets.X }} without pipe |
| T-06-02 | Tampering | Unauthorized workflow triggers | mitigate | paths: filter limits triggers; repo access controls apply |
| T-06-03 | Information Disclosure | Secrets in fork PRs | accept | GitHub does not expose secrets to PRs from forks by default |
| T-06-04 | Tampering | D1 SQL injection | accept | Migrations are static SQL files checked into repo, not user input |
| T-06-SC | Tampering | npm/pip/cargo installs | accept | No new packages added; workflow uses pre-existing wrangler from devDependencies |
</threat_model>
